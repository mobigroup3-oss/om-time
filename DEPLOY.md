# Деплой OM-Time на Vercel (хостинг + домен + бэкенд)

Сайт — статика без сборки (React + Babel из CDN). Vercel раздаёт файлы как есть
и автоматически поднимает функции из папки `api/`. Заготовки уже в репозитории:

- `vercel.json` — конфиг (чистые URL, редирект `/` → маркетинг).
- `package.json` — зависимость `@vercel/postgres` для функций (Build Command пустой).
- `api/` — функции: `booking.js` (приём заявок), `programs.js`, `team.js`,
  `schedule.js`, `hero.js`, `requests.js` (CRUD для кабинета), `_lib.js` (общие хелперы).
- `db/schema.sql` — схема БД; `db/seed.sql` — реальный текущий контент сайта.

**Как это работает.** Витрины (программы, команда, расписание, Hero) и редакторы
кабинета ходят в `api/` через `fetch`. Правка администратора пишется в Postgres и
видна **всем** посетителям. Если сервер/БД недоступны (например, файл открыт
локально без Vercel), страницы откатываются на встроенные данные (seed) — сайт не
ломается, но правки не сохраняются между браузерами.

---

## Шаг 1. Поднять сайт (5 минут)

1. Зарегистрируйся на **vercel.com** через GitHub.
2. **Add New → Project** → выбери репозиторий `mobigroup3-oss/om-time`.
3. Framework Preset: **Other**. Root Directory: **`/`** (репозиторий = папка `project/`).
   Build Command и Output Directory — **оставить пустыми**.
4. **Deploy**. Через минуту сайт открыт на `https://<имя>.vercel.app`.

С этого момента каждый `git push` в `main` деплоится автоматически.

## Шаг 2. Домен

- **`.com` (всё в одном на Vercel):** Project → **Settings → Domains → Buy** —
  покупка и привязка в одной панели, SSL автоматически.
- **`.kz` (отдельно):** купить на `ps.kz`/`hoster.kz`, затем в Vercel
  **Settings → Domains → Add** и прописать у регистратора DNS-записи, которые
  покажет Vercel (A / CNAME). SSL Vercel выпустит сам.

## Шаг 3. Включить бэкенд-функцию

Функция `api/booking.js` работает сразу (возвращает код брони). Чтобы заявки
сохранялись и приходили уведомления — добавь переменные окружения в
**Settings → Environment Variables** (любые из них, по необходимости):

| Переменная | Зачем |
|---|---|
| `POSTGRES_URL` | хранить данные сайта и заявки в БД (см. Шаг 4); подставляется автоматически при создании Postgres |
| `ADMIN_TOKEN` | **пароль администратора.** Защищает изменения (POST/PUT/DELETE) во всех `api/`. Без него запись в БД запрещена. Этот же пароль вводится на входе в `account.html` |
| `RESEND_API_KEY` + `NOTIFY_EMAIL` | письмо о новой заявке (resend.com, есть free-тариф) |
| `FROM_EMAIL` | адрес отправителя (после подтверждения домена в Resend) |
| `TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_ID` | уведомление в Telegram (бот через @BotFather) |

После изменения переменных нажми **Redeploy**.

> **Авторизация кабинета.** При входе в `account.html` запрашивается пароль —
> это значение `ADMIN_TOKEN`. Оно хранится в `sessionStorage` и шлётся в заголовке
> `x-admin-token` при каждом изменении. Это минимальная защита, а не полноценная
> пользовательская авторизация: один общий пароль на всех администраторов.
> Чтения (GET программ/команды/расписания/Hero) открыты; заявки (персональные
> данные) — только с токеном.

## Шаг 4. База данных

1. Vercel → **Storage → Create → Postgres** → привязать к проекту
   (переменная `POSTGRES_URL` подставится автоматически).
2. Вкладка **Query** → выполнить **`db/schema.sql`** (создаёт таблицы).
3. Затем выполнить **`db/seed.sql`** — наполняет БД реальным текущим контентом
   сайта (программы, команда, расписание, слайды Hero). Идемпотентно: повторный
   запуск ничего не дублирует. Без этого шага витрины покажут встроенный seed, а
   кабинет будет пуст до первого сохранения.
4. Драйвер `@vercel/postgres` уже объявлен в `package.json` — отдельных действий
   не нужно (Build Command остаётся пустым).

## Шаг 5. Форма записи и интеграция кабинета — уже подключены

В коде это уже сделано (отдельных правок не требуется):

- Форма записи (`BookingPage.jsx → handleSubmit`) шлёт `POST /api/booking`,
  показывает код брони из ответа и пишет копию в «Мои записи». Заявка появляется
  в разделе **Заявки** кабинета и (если заданы env) уходит письмом/в Telegram.
- Редакторы **Программы / Команда / Расписание / Карусель Hero** читают и пишут
  через `api/` с токеном; публичные страницы читают те же данные.

**Проверка сквозной связи:** войди в `account.html` (пароль = `ADMIN_TOKEN`),
измени программу/специалиста/событие/слайд → открой публичную страницу в другом
браузере и обнови → правка видна.

## Дальнейшие шаги (по мере надобности)

- **Полноценная авторизация** вместо общего `ADMIN_TOKEN` (отдельные аккаунты,
  роли, вход клиентов — сейчас «Мои записи»/«Профиль» живут в localStorage браузера).
- Слоты записи (`BOOKING_EVENTS`) и список программ воронки (`BOOKING_PROGRAMS`)
  пока остаются отдельным курируемым набором (свой namespace id под `api/booking.js`).
  При желании их можно связать с `/api/schedule` и `/api/programs`.
- Аналитика берёт число новых заявок из кэша `omtime.requests.v1`; можно перевести
  на прямой `GET /api/requests`.
